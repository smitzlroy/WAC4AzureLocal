Mid / Low-Level Design

Windows Admin Center for Azure Local at Global Scale

A pragmatic, Microsoft-aligned WAC topology for a regulated pharma manufacturing estate running many Azure Local instances per Azure subscription, optimised for ARM-first operations, OT segmentation, and disconnected-site resilience.

1. Executive summary

Recommendation: a two-tier WAC topology.

  • Tier 1 — Regional HA WAC (one HA pair per geo: e.g. US, UK, APAC). Active-passive failover-cluster install of Windows Admin Center, fronted by Microsoft Entra ID with Conditional Access & MFA. This is the daily-driver for the rare cases where ARM cannot perform the task.
  • Tier 2 — Per-site break-glass WAC. A single Windows Server VM on the site's existing management VLAN (not on the Azure Local cluster, not in Azure). Powered off by default; activated only when WAN/ExpressRoute to the regional WAC is unavailable, or when an OT control mandates fully-local management.

This pattern preserves the stated objective of as few WAC instances as possible without trading away local-site resilience or OT isolation. It validates the "≈3 per region" hypothesis as Tier 1 only, with Tier 2 added as a documented break-glass capability.

Recommended

Two-tier (Tier 1 + Tier 2)

Few servers, full local resilience, ARM stays primary, OT-friendly.

Valid pattern — weigh at scale

WAC-in-Azure VM

A supported and quick way to get started. As an estate grows multi-site and multi-region, several factors (latency, WAN dependency, network direction, blast radius) often favour an on-prem two-tier topology. See §3.1.

Preview for Azure Local

WAC in Azure portal (service)

Per Microsoft Learn, managing Azure Local cluster nodes via WAC in the Azure portal is documented as preview. Useful for opportunistic Arc-server tasks today; revisit once GA and at feature parity with the on-prem gateway.

OT note
This design assumes the OT control being invoked is "site-local management must remain available with no WAN dependency". If the actual OT control is different (e.g. "no inbound from corp at all", "no Entra ID at the site", "no domain trust"), the Tier 2 build changes. See §12 Open questions.

2. Terminology & gateway-mode disambiguation

Three distinct things are commonly called "gateway" in this space and have been mixed in prior threads. Lock these definitions down before any further design discussion:

TermWhat it actually isWhere it lives
WAC gateway server An installation mode of Windows Admin Center where the WAC .exe is installed on a Windows Server (or on a failover cluster for HA), and admins connect to it from a browser. This is what the HA deploy doc covers. On-prem Windows Server (single VM or 2-node cluster).
Azure Arc gateway for Azure Local An Azure resource that consolidates the outbound endpoints Azure Local nodes need to reach Azure. Reduces firewall allow-list size. Has nothing to do with WAC. Azure (resource) + Arc proxy agent on each Azure Local node.
WAC in Azure portal An Azure portal blade that lets you run a WAC-style UI against Arc-enabled servers / Azure Local nodes / Azure VMs without an inbound network path. Documented as preview for Azure Local cluster nodes. Azure portal (no on-prem server install).
Plain English
WAC gateway = the box you install. Arc gateway = an Azure-side traffic concentrator. They solve different problems and can coexist.

3.1 When WAC-in-Azure VM may not be the best fit at scale

Running WAC in an Azure VM is a fully supported pattern and a fast way to get started. As an estate grows to multiple sites and regions, several structural characteristics surface that can make an on-prem two-tier topology a better long-term fit. None of the points below are blockers — they are factors to weigh when planning beyond a pilot.

FactorWhy it can matter at scale
1. Management plane shares the WAN's fate WAC for Azure Local is most needed for tasks ARM cannot perform — often during incidents. If an incident coincides with ExpressRoute / S2S degradation, an Azure-hosted gateway is unreachable. An on-prem Tier 1 (with a per-site Tier 2 break-glass) keeps management local to the failure domain.
2. Latency on chatty WinRM / PowerShell sessions WAC operations are many small PowerShell remoting + WMI calls between the gateway and cluster nodes. Round-tripping each call from an Azure region back to a remote site adds tens to hundreds of ms; tools that issue many calls (volume operations, cluster validate, live-migration wizards) can feel sluggish or time out under load.
3. Direction of network flow Azure-hosted WAC initiates flows (WinRM 5985/5986, SMB 445, RPC ranges) from Azure into the site's management VLAN. On-prem WAC keeps that traffic east-west inside the site, which is generally easier for network and OT teams to approve and aligns with zone-separation principles (e.g. ISA/IEC 62443).
4. One Azure WAC across many regions concentrates risk A single Azure-hosted gateway managing US + UK + APAC clusters means one VM = one blast radius and forces every operator session through one geography. Per-region HA WAC distributes that risk and shortens the round-trip.
5. Authentication path during an Azure-side incident Azure-hosted WAC typically authenticates via Entra ID. During an Entra ID regional event, on-prem nodes remain healthy but the management UI is unreachable. On-prem Tier 1 can still use Entra ID; Tier 2 falls back cleanly to local AD or break-glass accounts.
6. Patching & certificate lifecycle is similar either way WAC is the same Windows Server install regardless of where the VM lives — same .exe, same cert handling, same upgrade caveats (e.g. 2311 → 2410+ uninstall/reinstall on HA). Hosting in Azure does not materially simplify lifecycle.
7. Don't conflate WAC-in-Azure-VM with WAC in Azure portal The Azure portal experience for managing Azure Local cluster nodes is documented as preview and is not yet at feature parity with a full WAC gateway. They are different things and should be evaluated separately.
Pragmatic adoption path
If you're already running WAC in an Azure VM, there's no need to retire it on day one. A typical sequence is: (1) stand up a Tier 1 HA WAC pair in one region as a pilot; (2) migrate that region's operators to it; (3) add a Tier 2 break-glass at one site; (4) repeat per region; (5) retire the Azure WAC VM, or repurpose it as an admin jump host.

5. Production pattern — Tier 1 HA WAC

5.1 Prerequisites

  • 2-node Windows Server failover cluster (Windows Server 2022 or 2025 recommended).
  • One Cluster Shared Volume (CSV), ~10 GB, for WAC persistent data.
  • TLS certificate from a trusted CA, with the private key installed on every node; capture the thumbprint.
  • WAC HA scripts: aka.ms/WACHAScript (Deploy-GatewayV2Ha.zip).
  • WAC installer (WindowsAdminCenter<version>.exe) staged on a node.
  • DNS A-record for the chosen client access point (e.g. wac-eu.contoso.local) and an optional static IP for the cluster generic service.

5.2 Sizing (starting point)

The HA install is active-passive per Microsoft Learn — one node serves traffic at a time and the other takes over on failure. This is not a load-sharing pair; size each node for full peak load.

ResourcePer nodeNotes
vCPU4 (suggested starting point)Modernised gateway is multi-process / micro-service; scale up for many concurrent admins.
Memory8–16 GB (suggested)Increase if many extensions are loaded.
OS disk127 GB (suggested)Standard Windows Server image.
CSV (shared)≥ 10 GBPer Microsoft Learn HA prerequisites.
Network1× mgmt NIC (+ optional cluster NIC)Same VLAN as the Azure Local management network or routed to it.

Microsoft Learn does not publish prescriptive vCPU/RAM minimums for the modernized gateway; values above are pragmatic starting points for review.

5.3 Install the HA gateway

From a node in the cluster, with the script and installer locally:

$parameters = @{
  ClusterStorage        = "C:\ClusterStorage\Volume1\Gateway"
  ClientAccessPoint     = "wac-eu"          # becomes https://wac-eu.<domain>
  StaticAddress         = "10.10.0.50"      # optional
  InstallerPath         = "C:\Installers\WindowsAdminCenter2511.exe"
  CertificateThumbprint = "AA11BB22CC33DD44EE55FF66AA77BB88CC99DD00"
}
.\Deploy-GatewayV2Ha.Deploy.ps1 @parameters

Validate and inspect with the sibling scripts (Deploy-GatewayV2Ha.Validate.ps1, Deploy-GatewayV2Ha.Inspect.ps1). To upgrade WAC later, re-run the same Deploy-GatewayV2Ha.Deploy.ps1 with the new -InstallerPath.

Upgrade caveat
Direct upgrades from WAC HA 2311 and older to 2410 and newer are not supported due to architectural changes in the modernized gateway. Plan an uninstall + reinstall when crossing that boundary. Source: Microsoft Learn (HA deploy guide).

5.4 Identity, RBAC & certificates

  • Identity provider: Microsoft Entra ID for Tier 1, with Conditional Access + MFA + (optionally) PIM-eligible roles.
  • Gateway roles: separate gateway administrators from gateway users.
  • Target-node RBAC: use WAC's role-based access control (JEA-backed) so daily operators are not full local admins on Azure Local nodes.
  • Certificates: use enterprise PKI; rotate via -UpdateCertificate on the same HA script. Avoid the -generateSslCert self-signed flag in production (60-day expiry).

6. Dev/Test pattern

One Windows Server VM (not a domain controller — installing WAC on a DC is unsupported). Local AD or Entra ID for auth. Self-signed cert acceptable for short-lived labs.

Build

  1. Provision Windows Server 2022/2025 VM (4 vCPU / 8 GB).
  2. Domain-join to the lab AD.
  3. Download WAC installer from the Microsoft Evaluation Center.
  4. Install in gateway server mode; bind to port 443.
  5. Add lab admins to the gateway administrators group.

Parity vs. Production

  • Same WAC tools and extensions are available.
  • No HA / no failover — single point of failure (acceptable for lab).
  • Use a different Entra ID app registration than production.
  • Do not connect lab WAC to production Azure Local clusters.

7. Per-site break-glass WAC (Tier 2)

One small Windows Server VM per manufacturing site, on the site's management VLAN (the same network segment from which the local Azure Local instance was deployed). It is a regular workload VM — not a cluster node, not hosted in Azure — and is powered off by default.

7.1 Where it lives

Two acceptable hosting options, in order of preference:

  1. As an Azure Local VM on the site's own cluster (recommended). It is a guest workload, not a cluster node. Survives WAN/ER loss because Azure Local continues to operate while disconnected from Azure (within the documented Arc grace period).
  2. On a dedicated small management host at the site, if one already exists. Useful if the site standard requires the management plane to be physically separate from the workload cluster.
Do not
Do not host Tier 2 in Azure, on the Azure Local cluster nodes themselves, on a domain controller, or on the OT process network.

7.2 Build

  • Windows Server 2022/2025, 2 vCPU / 4 GB / 127 GB. Domain-joined to the local site AD (or workgroup if AD is unavailable at the site).
  • WAC installed in gateway server mode on port 443. Enterprise CA cert preferred; otherwise self-signed with documented thumbprint trust on the operator workstation.
  • Local admin break-glass account documented in the site's existing privileged-access vault. Avoid hard dependency on Entra ID for the activation path.
  • Recommended: keep the WAC version within one release of the regional Tier 1 instances to minimise behavioural drift between tiers.

Why not just leave it on?
Leaving Tier 2 always-on doubles the attack surface per site and introduces drift between Tier 1 and Tier 2 RBAC. Powered-off-by-default keeps the OT segment quiet under normal operation and forces a deliberate, logged activation.

8.4 Arc gateway for Azure Local — orthogonal to WAC

The customer plans to use the Azure Arc gateway for Azure Local. This is unrelated to WAC topology but materially reduces the Azure-outbound endpoint list each Azure Local node needs. Key facts (Microsoft Learn):

  • Enable Arc gateway during deployment of new Azure Local instances on version 2506+. It cannot be enabled post-deployment today.
  • Arc gateway does not support TLS-terminating proxies.
  • A short list of endpoints (e.g. aka.ms, login.microsoftonline.com, management.azure.com, <region>.his.arc.azure.com, <arcgwid>.gw.arc.azure.com, Key Vault, Cloud Witness, CRL/OCSP) is not redirected and must remain in the firewall allow-list.
  • Azure Arc Private Link Scopes are not supported by Azure Local; Arc endpoints (*.his.arc.azure.com, *.guestconfiguration.azure.com, *.dp.kubernetesconfiguration.azure.com) must resolve to public IPs from the nodes / ARB / proxy.
OT / network — must read
HTTPS / TLS inspection is not supported for Azure Local outbound traffic. This includes Entra ID tenant restrictions v1. Coordinate with the OT/network team early — this is the single most common deployment blocker on regulated estates.

8.5 OT segmentation pattern (recommended)

  • WAC gateway servers (Tier 1 and Tier 2) sit in the same management VLAN as the Azure Local cluster's management network — never on the OT process network and never on the storage / RDMA fabric.
  • Operator browser traffic enters via a documented jump path (e.g. PAW or VDI) → Tier 1 WAC VIP only.
  • Tier 2 has no inbound from corp WAN; only the local site PAW reaches it.
  • All east-west WAC→node traffic stays inside the site's mgmt VLAN; only Azure-outbound traffic (443) traverses the site firewall, ideally via the Arc gateway.

10. Operations & lifecycle

  • Patching: WAC non-preview versions are supported until 30 days after the next non-preview release. Stay within one release of current.
  • Version skew: 2311 → 2410+ requires uninstall + reinstall on HA. Plan a maintenance window.
  • Cluster Manager extension: ensure version ≥ 5.2.6 on every WAC instance before performing volume operations on Azure Local; otherwise update to WAC 2511 build 2.6.6.18 or higher (or use WAC-in-Azure-portal extension 0.76.0.0+). Source: WAC known-issues, April 2026 update.
  • Tier 2 drift control: quarterly job to power on each break-glass VM, patch, validate, power off. Track in change management.
  • Backup: WAC HA stores config on the CSV — back up the CSV and the cert. Tier 2 is stateless enough that re-imaging is the recovery path.

11. VM console / out-of-band access

WAC does not solve the "I need a console when WAC itself is broken" problem. Treat this as a separate, parallel capability and document it in the same runbook:

  • OEM BMC (iDRAC / iLO / XClarity) on a dedicated, isolated mgmt VLAN, reachable only from a site PAW.
  • Hardened jump host per site for serial-over-LAN / KVM access.
  • Document MFA and credential rotation for BMC accounts; these are typically the most-overlooked privileged accounts in pharma OT estates.

12. Open questions for the customer

We need answers to finalise Tier 2
  1. Which specific OT control is being invoked when you say "OT security requires X"? Is it ISA/IEC 62443 zone separation, a site-specific change-control rule, or a regulatory interpretation? The Tier 2 build differs for each.
  2. Does each site have its own AD, or is AD homed regionally? This decides how Tier 2 authenticates with no WAN.
  3. Confirm ExpressRoute / WAN design: single-circuit per site, or dual? Tier 2 only justifies its existence under realistic single-failure scenarios.
  4. Confirm enterprise PKI source for WAC certificates and rotation cadence.
  5. Confirm site count per region (drives the Tier 2 fleet size and patch rota).
  6. Confirm whether Arc gateway will be enabled at initial Azure Local deployment (it cannot be added later).
  7. Confirm operator workstation strategy (PAW vs. VDI vs. corp laptop) — affects browser-side firewall rules.
